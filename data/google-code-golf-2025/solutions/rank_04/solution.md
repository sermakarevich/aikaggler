# Parallel Sampling

My core strategy was to have an LLM (`gpt5-codex`) perform code golf using tools in parallel. gpt5-codex has very high tool utilization capabilities, and was able to analyze JSON test cases in a REPL and do various things on its own. For each task, we would run a single conversation session consisting of 10-30 turns. This entire process was repeated many times every day. The duration of a single turn varied depending on the problem's difficulty, ranging from 5 minutes to over an hour.

The figure below illustrates one "turn" in a session. It begins with a prompt and generated 4 parallel versions. Then, each generated code is evaluated, and the shortest valid code is selected. Code golfing continues with a follow-up prompt for the next turn.

# Rule-based Prompt Generation

Through experimentation, I found that providing the LLM with specific, concrete ideas for improvement was effective. The base follow-up prompt was simple:

```
"Please make it as short as possible in bytes for Code Golf and tell me the file size."
```

On top of this, I generated additional rule-based prompts using the code's AST. For example:

- If the code didn't use regular expressions, I added: "Try to use the regex re package to code golf. It might be a key to a breakthrough."
- If the code used loops, I instructed it to try converting them into recursive functions.
- If a function was defined with def, I prompted it to use lambda. While not always optimal, this was often the better choice.

I co-developed the AST-based prompt generation script with `grok4-code-fast-1` and `gpt5-codex`. If I had embedded more domain-specific code golf knowledge into these rules, the speed of improvement could have been even faster.

# Stop Criterion & Exploration

I terminated a conversation session if the score failed to improve for $s$ consecutive turns. If there was no improvement for $t$ consecutive turns (where $t$ < $s$), I would prompt the model to be more exploratory with instructions like:

```python
"Analyze the test cases generation code."
"Rethink a more compact algorithm."
# "Focus on reducing the number of bytes, even if it means sacrificing some functionality."
```

I stopped using the third, commented-out prompt as I found it wasn't very effective.

# Escaping Local Optima

When using Codex Cloud (Web), the model had access to the repository containing the previously generated code. This often caused it to get stuck in a local optimum, unable to break away from the existing solution.

To solve this, when using codex exec (CLI), I created a clean, empty sandbox environment for each new session. This forced the model to start from a blank slate. I remember breaking out of several difficult local optima on a few tasks using this method.

I also tried to achieve this in Codex Cloud (Web) by explicitly instructing the model to start from scratch. It wasn't perfect, but I think it worked. The prompt included a line like:

```
"Ignore the code that's already implemented and write it from scratch."
```

# LLM optimize for "compression-friendly" code

In the last 2 weeks, I managed to significantly improve our scores by having the LLM optimize for "compression-friendly" code. The idea was to generate code that would be highly compressible with zip(zopfli). I used the same Parallel Sampling framework but changed the evaluation function to score based on the post-compression file size. The prompt for this was:

```python
"This must be achieved with minimal Kolmogorov complexity and is evaluated based on the file size after zip compression."
```

I thought this was a unique improvement that other teams might not have tried, but in the end, their code was often much shorter, which was a bit disappointing. I did have one 💎 solution where this technique worked exceptionally well: Task 324. You can see our compressed solution here: https://gcgc.pages.dev/task324.
