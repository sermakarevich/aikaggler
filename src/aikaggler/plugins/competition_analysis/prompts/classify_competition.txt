You are classifying a Kaggle competition into a structured taxonomy. Input is JSON metadata from Kaggle (title, description, evaluation metric, categories, etc.) plus a sample of top-solution titles. Produce a classification following the schema.

Fields:

- 'description': 2-4 sentences in plain English describing what the competition is about, what data competitors work with, what they need to predict, and how success is measured. Written for a reader who has not seen the competition page. Do NOT copy the title verbatim; synthesize. Example: 'Participants classify protein subcellular localization in fluorescent microscopy images into 28 classes. Each image has up to 4 stained channels and can show multiple proteins at once (multi-label). Submissions are ranked by macro F1 on a held-out private test set.'

Taxonomy — 3 levels:

LEVEL 1 — what the competitor works with:
- 'data_modality': the primary data type. If truly mixed (e.g. image + text together in a single model), use 'multimodal'. Otherwise pick the dominant one.
- 'task_type': the core ML task. 'classification' covers binary / multi-class / multi-label at the head level; use 'segmentation' / 'detection' for pixel or bbox outputs; 'metric_learning' when the objective is embeddings / similarity; 'forecasting' when targets are future time-steps; 'generation' for free-form outputs (image / text / audio generation).

LEVEL 2 — drivers of the winning-solution shape:
- 'label_structure': 'single_label' (one class per example), 'multi_label' (multiple labels possible), 'hierarchical' (taxonomy), 'none' (unsupervised / self-supervised), or 'unknown'.
- 'metric': copy the evaluation metric verbatim (e.g. 'F-Score (Macro)', 'Quadratic Weighted Kappa', 'Mean Average Precision @ K=5', 'RMSE').
- 'test_split': how train/test are split. 'temporal' if test is a later time window (forecasting, user-log comps); 'grouped' if splits respect a group id (by user, by patient, by image_id); 'stratified' if per-class balanced; 'random' if iid random; 'unknown' if not derivable.
- 'format': 'code_comp' = notebook-only submission with compute budget; 'two_stage' = separate public/private test phases; 'simulation' = agent/RL env; 'standard' otherwise.
- 'dataset_scale': rough order of magnitude. 'small' <1GB, 'medium' 1-100GB, 'large' 100GB-1TB, 'xlarge' >1TB. 'unknown' if not stated.
- 'domain': free text describing the application area (e.g. 'medical_imaging', 'retail_forecasting', 'autonomous_driving', 'genomics', 'ads_ctr', 'nlp_qa'). Prefer snake_case, 1-3 words.

LEVEL 3 — challenges / constraints (best-effort; leave [] if nothing obvious):
- 'data_challenges': data-side difficulties the competition is known for. Examples: 'class_imbalance', 'label_noise', 'domain_shift', 'leaky_features', 'duplicate_images', 'small_test_set', 'few_shot_classes', 'long_tail', 'private_test_shift', 'unlabeled_data_abundance'.
- 'constraints': submission / resource constraints. Examples: '9h_inference_limit', 'no_internet_during_inference', 'no_external_data', 'no_pretrained_models', 'kernels_only', 'tpu_only', 'single_gpu'.

Rules:
- Use ONLY what is stated or strongly implied by the metadata + solution titles. Do not invent domain-specific jargon.
- If a field is ambiguous, pick 'unknown' or 'other' (for enums) or [] (for lists) rather than guessing.
- Kaggle's own 'categories' field (e.g. 'data type > image', 'task > classification') is a strong hint — trust it for L1.
